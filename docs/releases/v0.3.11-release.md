# Presto 0.3.11

`0.3.11` 覆盖 `v0.3.10` 之后的全部变更。这个版本的重点是把 PTSL 命令最低宿主版本收口为统一元数据，恢复插件工作流可用性，并把 Pro Tools PTSL 适配更新到 `PTSL_SDK_CPP.2026.04.0.1301892`。

## 重点变化

- 应用、workspace package、Tauri、Rust crate、contracts、SDK runtime 与 backend 版本基线统一到 `0.3.11`。
- PTSL catalog 不再向外暴露 `introducedVersion`，统一使用 `minimum_host_version` / `minimumHostVersion` 表达“执行该命令所需的最低 Pro Tools/PTSL 宿主版本”。
- 新增集中配置 `backend/presto/integrations/daw/ptsl_requirements.json`，统一维护 PTSL 命令最低版本覆盖和 composed capability 的命令依赖，避免版本要求散落在 adapter、manifest 和前端判断里。
- 插件宿主现在可以根据插件 `requiredCapabilities` 自动读取当前 DAW capability snapshot 中的 `minimumHostVersion`，取最高值作为插件在当前 DAW 下的最低宿主版本需求。
- 官方插件 manifest 移除手写 capability version requirement，改由 host 根据 capability 自动推导最低宿主版本需求。
- Pro Tools PTSL 适配更新到 2026.04 SDK：catalog 当前包含 153 条命令，并同步到 backend capability catalog、contracts capability registry 与 generated capability id 列表。
- 已安装扩展列表与 automation 卡片现在都会显示当前插件在当前 DAW 下推导出的最低版本要求，例如 `Pro Tools ≥ 2025.10.0`。
- app 启动和手动刷新 DAW 状态时会先做短超时连接探测，再读取 DAW status，减少 unknown、disconnected、connected 逐次跳变的情况；后台定时轮询仍只读 status。

## PTSL 最低版本模型

- `scripts/generate-ptsl-catalog.mjs` 会从 SDK `PTSL.proto` 的 `@since Pro Tools ...` 注释提取每条命令的初始最低版本。
- 对 SDK 注释缺失或当前业务需要修正的命令，生成脚本读取 `ptsl_requirements.json` 中的 `commandMinimumHostVersionOverrides`。
- `ptsl_requirements.py` 提供 `ptsl_commands_for_definition()` 和 `minimum_host_version_for_definition()`，host / backend 可从 capability definition 自动反查其 PTSL 命令依赖，并计算最高最低版本。
- `daw.adapter.getSnapshot` 返回的 capability snapshot 增加 `minimumHostVersion`，前端和 SDK runtime 使用同一字段做插件可用性判断。
- composed workflow capability 不再靠插件 manifest 自己写版本要求；例如导入、导出和 split stereo automation 会从它们使用的 PTSL 命令集合自动计算最低版本。

## 插件可用性与前端展示

- 插件可用性判断现在同时考虑：
  - `supportedDaws` 是否覆盖当前 DAW。
  - adapter module requirement 是否满足。
  - capability requirement 是否满足。
  - 插件 `requiredCapabilities` 对应 capability 的 `minimumHostVersion` 是否高于当前 host version。
- Workflow Extensions / Automation Extensions 设置页会在每个插件条目上显示当前 DAW 的最低版本要求；展开详情时也会显示同一结果。
- Automation 首页卡片会显示同一最低版本要求。该值由 host 在过滤 automation entry 时注入，卡片本身不重复理解插件 manifest 或 DAW snapshot。
- Tool 插件不绑定 DAW，不显示 DAW 最低版本要求。

## Track Pan 处理

- `daw.track.pan.set` 原先依赖 `CId_SetTrackControlBreakpoints` 写 pan automation breakpoint；该命令在 Pro Tools 26.4 SDK 中已删除。
- 当前版本不再把 track pan 当作 PTSL-backed command 暴露，而是改为现有 UI automation 能力：
  - 校验 `trackName` 和 `value`。
  - 通过 DAW adapter 选择目标轨道。
  - 执行 Accessibility preflight。
  - 执行 DAW UI profile 的 set track pan 脚本。
- 直接调用 `ProToolsDawAdapter.set_track_pan()` 不会再尝试发送已删除命令，而是返回明确的 `TRACK_PAN_UNAVAILABLE`，提示使用 capability/UI automation 路径。

## py-ptsl 26.4 schema 缺口

- 当前环境中的 `py-ptsl` 版本仍是 `601.0.0`，其 `PTSL_pb2` 尚未包含 Pro Tools 26.4 新命令的 request / response message。
- `PtslCommandRunner` 现在按两种路径执行：
  - Python schema 存在时，继续使用 protobuf JSON parse 做 request 校验和 response normalize。
  - catalog 中存在但 `py-ptsl` 尚无 schema 的 26.4 新命令，按 command id 直发 raw JSON，并原样返回 response。
- PTSL coverage 报告会明确列出这类 unresolved schema，避免把 SDK 已有命令误判为 catalog 错误。

## 修复

- 修复插件版本要求收口后，官方导入、导出和自动化插件可能因为重复的手写 capability version requirement 被错误判定不可用的问题。
- 修复 PTSL command execute 路径中最低版本字段命名不一致的问题；request payload 与 snapshot 输出现在使用统一语义。
- 修复 26.4 SDK 下 `daw.track.pan.set` 仍试图走已删除 PTSL command 的风险。
- 修复启动和手动刷新路径只读取 `daw.connection.getStatus` 而不建立 PTSL 连接，导致连接状态可能停留在 unknown/disconnected、需要多次刷新或其它连接动作后才显示已连接的问题。
